# PLAN — Fleet hierarchy reorganization (manageability)

Status: DRAFT for Codex adversarial review · 2026-06-09

## The real problem (not the literal ask)

Stated: "move the fleet folders in `~/src` into a hierarchical structure inside issen … group ewf/aff4/vmdk/vhdx inside a container folder … I can't remember how many container formats I have planned."

Decomposed, there are **two distinct needs**:
1. **Discoverability** — a mental model: *which* formats exist, which are shipped vs planned, grouped by layer. ("how many container formats planned.")
2. **Folder ergonomics** — a browsable, grouped directory tree instead of a flat 60-repo dump.

These have very different cost/risk profiles and should be solved separately.

## Hard constraints (what makes the literal move dangerous)

- The fleet is **~25 independent git repos**, most **published to crates.io + GitHub** (ntfs-core 0.7.1, ewf, vmdk-core, forensicnomicon, …), each with its own CI, release tags, gitsign, `docs/` site.
- **31 genuine cross-repo `path = "../x"` dependency lines** (Codex-verified; the earlier 68 conflated intra-workspace `../core`/`../forensic` deps which are unaffected by moving a whole repo). Concentrated in **5 repos: issen (23), 4n6mount (4), ext4fs-forensic (2)**, plus memory-forensic→winevt-forensic. These assume a flat sibling layout and break the moment repos move to different depths.
- **CI workflows already hard-code flat sibling checkouts** (Codex-found): `ext4fs-forensic/.github/workflows/ci.yml:33` checks out `../4n6mount`+`../ewf`; `4n6mount/ci.yml:37`, `browser-forensic/ci.yml:29` (`../forensicnomicon`), `srum-forensic/ci.yml:26`. A physical move breaks these too — they must be repointed (or the deps registry-ized) before any move.
- **Immune to a *local* move** (only a GitHub *rename* breaks them): `repository =` fields, docs.rs backlinks, `securityronin.github.io/<repo>/` footer URLs, homebrew taps (`homebrew-blazehash`, `homebrew-nameback`), `.git/config` remotes. A folder move keeps each `.git` + remote intact.
- Hardcoded `~/src/<repo>` absolute paths live in CLAUDE.md, plan docs, the corpus catalog, `/tmp/case001` staging notes, and (likely) IDE/workspace configs and shell history.
- **"Inside issen" is a trap if taken literally:** `~/src/issen` is itself a git repo (a crate workspace). Moving other git repos *into* it creates nested repos — issen's git would see them as untracked dirs or accidental submodules. **Never physically place fleet repos inside the issen crate's git tree.**
- The user's own binding rule: **prefer published registry crates over path deps.** The end-state where all cross-repo deps are *registry* (version-based, location-independent) makes physical layout purely cosmetic — moving a repo then breaks nothing.

## Options

### A — Index only (zero physical change) — solves need #1
A maintained `issen/FLEET.md` (or `docs/fleet.md`) generated by a small `fleet` script that scans `~/src`, classifies each repo by layer, and emits a hierarchical table: **layer → repo → crate(s) → status (shipped vX / planned / stub) → crates.io link → one-line purpose.** "How many container formats planned" becomes one glance.
- Pros: zero risk, zero breakage, reversible (a doc), directly answers the stated mental-model gap, regenerable.
- Cons: does not change the on-disk folder layout (need #2 unmet).

### B — Symlink view over unmoved repos — solves #2 with near-zero risk
Real repos stay flat at `~/src/<repo>` (all 68 path deps and every `~/src/<repo>` ref keep working). A generated tree `~/src/fleet/container/ewf -> ~/src/ewf`, … gives the browsable hierarchy.
- Pros: full grouped tree, **zero dependency/ref breakage**, trivially reversible (`rm -rf` the view), regenerable from the index.
- Cons: symlinks can confuse some tools (cargo path-dep resolution through links, IDE indexers, `git` if opened via the link); two paths to the same repo.

### C — Physical move into `<umbrella>/<layer>/<repo>` — the literal ask, highest cost
Each repo physically moves (its `.git`/remote travel with it — independent identity preserved). Requires: fix all cross-repo path deps, update every hardcoded `~/src/<repo>` ref, update IDE/workspace/scripts.
- Pros: the real grouped tree the user pictured.
- Cons: breaks 68 path-dep lines + N hardcoded refs at once; high churn; disruptive to in-flight work (Case 001 staging, the SQLite/br4n6 plan's `~/src/...` refs). Reversible only via a saved move-manifest script.

### D — Git submodules under an issen super-repo — rejected
Hierarchical + independent repos, but submodules impose detached-HEAD / double-commit / sync friction across 25 *actively co-developed* repos. Wrong tool for this dev style.

## Codex verdict (2026-06-09)

> **GO: Hybrid A+B — `FLEET.md` index + a `~/src/fleet/` symlink view as a *browse-only* layer.** Near-zero risk; solves the stated problem immediately. **Do NOT gate A+B on registry migration.** The symlink view is sound for navigation/indexing but **unsound as a Cargo/rust-analyzer entrypoint** — Cargo does not canonicalize symlinks for nested `../sibling` path deps, so `cargo`/IDE must always be invoked from the **real** `~/src/<repo>` path, never through the symlink. Registry-dep migration (Phase 2) gates **only** the optional physical move (Phase 3). For an eventual move, prefer **`~/src/4n6/`** over demoting `issen` (option C creates needless churn relocating the active issen workspace + its internal `crates/issen-*` members).

## Recommended: phased hybrid (A → B → optional C, de-risked)

1. **Phase 0 — the map (now, zero risk):** build `FLEET.md` + the `fleet` classifier script. Solves need #1 immediately. Encodes shipped-vs-planned so "how many container formats planned" is answered and stays answered.
2. **Phase 1 — the view (low risk):** generate the `~/src/fleet/<layer>/` **symlink** tree from the index. Solves need #2 with no breakage. Validate cargo still builds through the real paths (it does — repos unmoved).
3. **Phase 2 — de-risk deps (the precondition for any real move):** drive cross-repo **path deps → registry deps** (publish the remaining fleet crates: winevt-*, memf-*, browser-*, srum-*, …). When path deps ≈ 0, physical location is cosmetic.
4. **Phase 3 — optional real move (only after Phase 2):** if the symlink view proves insufficient, do a **scripted, reversible** physical move into `<umbrella>/<layer>/`, fixing the now-tiny residual path-dep set + a grep-driven hardcoded-ref sweep, with a saved inverse manifest.

`<umbrella>` and "inside issen" need a decision (see below); the move never goes *inside the issen crate's git tree*.

## Reversibility & safety (binding for any physical change)
- Every move is emitted as a manifest (`from → to`) with an auto-generated inverse script.
- No move proceeds while a repo has uncommitted changes or unpushed commits.
- After a move: `cargo metadata` on every workspace must resolve; CI must stay green per repo.
